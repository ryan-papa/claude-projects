# Harness: 디자인 원칙

> UI가 있는 프로젝트(웹/앱)에 적용. 토스 디자인 철학 기반.

## 1. Simplicity

- 한 화면 한 목적. 사용자가 배우지 않아도 직관적으로 이해
- 기능 추가는 최후의 선택 (Minimum Features)
- 정책/조건이 복잡하면 UI로 극복 불가 → 정책 자체를 단순화

## 2. 컬러 시스템

**시맨틱 토큰 구조:**
```
Primitive (원시값) → Semantic (의미 부여) → Component (적용)
```

**필수 시맨틱 컬러:**

| 용도 | 역할 |
|------|------|
| Primary | 주요 액션, CTA 버튼 |
| Success | 성공 피드백 |
| Error | 에러, 위험 |
| Warning | 경고, 주의 |
| Disabled | 비활성 상태 |
| Background | 배경 |
| Text / Text-light | 본문 / 보조 텍스트 |

- CSS 변수로 관리 (다크모드 대응 가능)
- 프로젝트 특성에 맞는 팔레트 선택 (아이용 = 밝고 컬러풀, 비즈니스 = 깔끔 미니멀)

## 3. UX 라이팅

| 규칙 | 내용 |
|------|------|
| 문체 | 해요체 통일 |
| 문장 | 능동형, 긍정형 우선 |
| 길이 | 핵심 메시지 한 줄 |
| 단어 | 모든 사용자가 이해하는 단어 (전문 용어 지양) |
| 톤 | 제안형 (강요/공포감 금지) |
| 버튼 | 다음 화면에서 일어날 일을 예측 가능하게 |

**에러 메시지 원칙 (Navigating Error):**
- 무슨 일이 일어났는지
- 왜 일어났는지
- 어떻게 해결할 수 있는지
- 사용자 탓이 아닌 시스템 관점으로

## 4. 인터랙션

| 항목 | 원칙 |
|------|------|
| 전환 | 매끈하고 자연스러운 애니메이션 |
| 터치 | 시각적 피드백 (색상 변화, 스케일) |
| 로딩 | 사용자를 기다리게 하지 않음 (No More Loading) |
| 피드백 | 성공/실패 즉시 시각적으로 전달 |

## 5. 접근성

- 터치 타겟 최소 48px
- 텍스트 대비비 충분히 확보
- 디지털에 익숙하지 않은 사용자도 사용 가능하도록
- 질문형 버튼명 지양 (혼란 유발)

## 6. 컴포넌트

- 재사용 가능한 컴포넌트 단위로 설계
- 일관된 스타일 (같은 버튼은 어디서든 같은 모양)
- 점층적 구조: 타입 → 내부 요소 → 상세 스펙
