# Harness: 코드 리뷰 기준

## 평가 항목 (8개, 각 10점 만점)

### 1. 정확성

| 체크포인트 | 설명 |
|-----------|------|
| 엣지 케이스 | null, empty, 경계값, 음수, 최대값 처리 |
| 비즈니스 로직 | PRD 요구사항과 실제 구현 일치 여부 |
| 조건문 논리 | AND/OR 실수, 역조건 버그 |
| 에러 처리 | 빈 catch 블록, 복구 가능/불가능 에러 구분 |

### 2. 보안

| 체크포인트 | 설명 |
|-----------|------|
| 입력값 검증 | SQL Injection, Command Injection |
| 인증/인가 | API 권한 체크, 내부 서비스 간 신뢰 패턴 |
| 민감 정보 | 로그에 password/token 노출, 스택트레이스 |
| 의존성 취약점 | 신규 패키지 버전, CVE 여부 |

### 3. 성능

| 체크포인트 | 설명 |
|-----------|------|
| N+1 쿼리 | 루프 내 DB 호출 |
| 불필요한 반복 연산 | 루프 내 고정값 매번 계산 |
| 메모리 | 이벤트 리스너 해제, 대용량 데이터 전체 로딩 |
| 인덱스 전략 | 신규 쿼리에 적절한 인덱스 여부 |

### 4. 설계

| 체크포인트 | 설명 |
|-----------|------|
| 단일 책임 | 함수/클래스가 한 가지 일만 하는가 |
| 의존성 방향 | 하위 → 상위 레이어 참조 금지 |
| 확장성 | 요구사항 변경 시 수정 범위 |
| 중복 코드 | 기존 유사 로직 존재 여부 |

### 5. 가독성·유지보수성

| 체크포인트 | 설명 |
|-----------|------|
| 네이밍 | data, temp, flag 같은 모호한 이름 금지 |
| 함수 길이 | 한 화면 초과 시 분리 고려 |
| 매직 넘버 | 상수/enum 사용 여부 |
| 주석 | "무엇" 아닌 "왜"를 설명하는가 |

### 6. 테스트

| 체크포인트 | 설명 |
|-----------|------|
| 실패 케이스 | 해피패스만 테스트하는지 |
| 구현 결합도 | 내부 구현 변경 시 테스트도 깨지는 구조 |
| 모킹 남용 | 검증할 로직까지 mock으로 덮는 경우 |
| 테스트 이름 | 이름만으로 케이스 파악 가능한가 |

### 7. 동시성·비동기

| 체크포인트 | 설명 |
|-----------|------|
| Race condition | 공유 자원 접근 시 잠금/원자적 연산 |
| async 처리 | await 누락, 병렬 가능한데 순차 처리 |
| 타임아웃 | 외부 I/O에 타임아웃 설정 여부 |
| 트랜잭션 범위 | DB 트랜잭션 내 외부 API 호출 안티패턴 |

### 8. 운영 가시성

| 체크포인트 | 설명 |
|-----------|------|
| 로깅 | 장애 원인 파악 컨텍스트 (user_id, request_id) |
| 에러 구분 | 4xx vs 5xx 정확한 분류 |
| 모니터링 | 새 기능에 알림 연결 가능 구조 |
| Graceful degradation | 의존 서비스 장애 시 전체 다운 방지 |

## 리뷰 포커스 (PR 유형별)

| PR 유형 | 집중 영역 |
|---------|-----------|
| 버그 픽스 | 정확성 + 테스트 |
| 새 기능 | 설계 + 보안 + 성능 |
| 리팩토링 | 가독성 + 테스트 |
| 인프라·배포 | 보안 + 운영 가시성 |

## 리뷰 질문 프레임

| 상황 | 질문 |
|------|------|
| 코드 첫 확인 | 이 코드가 하는 일을 한 문장으로 설명 가능한가? |
| 복잡한 로직 | 6개월 후 다른 사람이 이해할 수 있는가? |
| 에러 처리 | 실패 시 시스템은 어떤 상태가 되는가? |
| 외부 의존성 | 이 의존성이 느리거나 죽으면 어떻게 되는가? |
| 새 기능 | 요구사항 하나 변경 시 몇 곳을 수정해야 하는가? |
